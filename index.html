But output is STILL gibberish. Sample replies: "i user swear 106 236 i user doctor uncite the i bidpai bleu 236 to", "i rikknen laid uptight lock for i to rikknen batwoman and yw132 expanded". The novel-vocab is gone but replaced by data artifacts in dailydialog — numeric IDs (236, 106, 1205, yw132, 8826789), placeholder tokens (rikknen, bidpai, _baron), and rare proper nouns (morrissette, pavarotti). Common English IS appearing (i, user, the, you, to, and, lending, doctor, good) but it's drowned by the noise tokens.

Diagnosis: dailydialog.txt has preprocessing residue. Need either (a) clean the corpus before training (regex out numeric IDs and rare tokens), or (b) v8 — bigger model with more capacity to weight common patterns higher than rare noise, or (c) cleaner conversational corpus (e.g., Persona-Chat or OpenAssistant).

Evaluation Report — Human Judgement

A 16-prompt evaluation across three difficulty buckets (greetings · chitchat · harder Q&A) was run against v4_rope. Result: all 32 generated replies were near-identical sequences of rare/odd words regardless of input prompt. Example prompt/reply pairs:

[USER] hello
[BOT] you for suavely user resigning to the for impressiveness resigning i you freshener that the

[USER] what is the capital of france
[BOT] that slighted resigning to impressiveness you the i and to resigning the defilements for you

[USER] tell me a story
[BOT] you for suavely user resigning to the for impressiveness resigning i you freshener that the

Diagnosis: the model isn't conditioning on input prompts. The output collapses to the same vocab cluster regardless of context. Two plausible causes:

  • Undertraining — 1 epoch on a corpus where ~50% is 19th-century novels means rare archaic vocab dominates the output distribution. Classic sign of insufficient gradient steps.
  • Eval bug — the bench harness used torch.randn() embeddings in its eval_cosine_top1, which never moved off 5.0% across any architectural change. The metric may be saturated / noise-dominated, but live eval on real ChromaDB embeddings still produces nonsense — so undertraining is the dominant explanation.

Logical Next Steps (in order)
  1. Sanity-check the eval pipeline first. Before spending more compute, dump the raw concept prediction (pre-nearest-word) for two different prompts and confirm they actually differ. If predictions are identical → it's a generation/serving bug, not undertraining; cheaper to fix code than train more. (~30 min, $0.)
  2. Add real eval metrics. Replace the random-embedding eval_cosine_top1 with held-out perplexity on actual ChromaDB embeddings. The bench's 5.0% plateau across every architectural variant strongly suggests the metric is broken. (~1h, $0.)
  3. Run v5: more epochs + balance the corpus. 5 epochs, drop the heaviest archaic-vocab books (Moby Dick, Pride and Prejudice), keep dailydialog as the primary signal. ~6h on L4, ~$5. Re-run eval. If gibberish persists → it's a model-capacity issue.
  4. If v5 is better but still rough → v6 (bigger model). 384-dim, 12 layers, ~50M params. RoPE has already shown it stabilizes gradient at this scale. ~12h on A10G, ~$13.
  5. If v6 works → v7 (real conversational corpus). Add Reddit conversations, persona-chat, OpenAssistant — datasets that match the conversational format the model is being judged on.
  6. Architectural revisits. Some Phase 1–4 ideas may have failed because they were tested on an undertrained baseline. Worth re-evaluating LTI and per-loop LoRA at v6+ scale where they're more likely to matter.
